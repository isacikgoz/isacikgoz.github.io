New Roadmap of Gitin

İbrahim Serdar's Blog
2019-03-01

There have been approximately a month since I started to develop gitin. For a quick reminder, gitin is a commit/branch/workdir explorer for git. The tool itself brings a handy and minimalist interface to git browsing. I advertised it with its log browsing feature. You can simply run gitin log and start exploring commits. It is an interactive alternative to git log which allows you to explore individual files in the commits and vice versa. You can also filter commits for such as date, author, ahead, behind and so on. De facto usage of gitin log is something like below:

The initial thought for creating gitin has emerged from the idea of “an interactive alternative to git status“. But, somehow, the interactive log has come to my focus and I implemented it and put that feature to focus. I also use gitin for a month and I actually find irritating using of gitin log. It is not that good for my daily usage. It is distracting and not so intuitive. Don’t get me wrong, I use gitin every day, but not for logging, I use it for gitin status. Just as I imagined it. The gitin status command is to replace the workflow of git status, git add and git commit. It works better than expected for me. I love using it and it is very powerful for exploring the current status of my repository. It also allows built-in interactive hunk staging so that I can stage, commit individual diff hunks. It saved a lot of time of mine. The demo can be found below:

Almost two weeks after I published gitin I met with my friend Fatih to eat dinner. Fatih brought his son Alper, and I brought my son Erdem and as a result, we couldn’t have a clean conversation because these toddlers are nasty! Even though there was a little window to talk about our side-projects. Fatih told me about an idea of better usage of gitin log. That was a fast fuzzy finder for git. In the beginning, I bragged, come’on bro, gitin already supports fuzzy search. Just type / and enter your text and let the tool find results in real-time. Actually, Fatih was also aware of that feature but his idea was to make commit browsing intuitive. He said, “I just want to type something e.g. a part of commit msg or a branch name and the tool would bring me that.” The user experience should be straightforward. Once I execute the tool, I should start typing. Then, I added a flag to allow that like gitin log --search would be a solution. And you can alias it to let’s say gls and you would be fine. But, no it was too much configuration and Fatih and I both agreed on this would not improve the user experience. If this is a better approach for exploring commits, it should be the default. Also, when you are searching for commit you may want to see more options. Plus, you may want to see which text matched with the result. So, the feature was evolving to a new look and feel. Since I am at the start of developing this tool, I think I can make some radical changes.

And it all started here. The radical changes. My current decision is to split gitin into two different tools. One for status, which is the fundamental notion of gitin. And one for the “interactive commit/branch exploring”. Here is the proposal:

  • brand new sig tool which has basically the same behaviour with gitin status but has a brand new implementation.
  • brand new fig tool which covers both gitin log and gitin branch functionality and presents a more intuitive user experience.
